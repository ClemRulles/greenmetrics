## Onboarding Flow (Post-PR #14)

### Admin Workflow
1. **Create Invitation**: `POST /api/orgs/{orgId}/invitations` (ADMIN+ required)
   - Email automatically sent with acceptance link
   - 14-day TTL (configurable via INVITE_TTL_DAYS)
   - Bilingual support (EN/FR) based on locale parameter

2. **Monitor Invitations**: `GET /api/orgs/{orgId}/invitations` (ADMIN+ required)
   - List pending invitations
   - Track expiration status

3. **Revoke if Needed**: `POST /api/invitations/{token}/revoke` (ADMIN+ required)
   - Immediately invalidate invitation
   - Audit trail maintained

### User Workflow
1. **Receive Email**: User gets invitation email with secure acceptance link
2. **Sign In/Up**: User must authenticate with same email address
3. **Accept Invitation**: `POST /api/invitations/{token}/accept`
   - Email validation enforced
   - Automatic membership creation with specified role
   - Invitation deleted after acceptance

### Operational Notes
- **Email Config**: Uses EMAIL_FROM and EMAIL_SERVER; dev mode logs to console
- **Security**: Tokens are 24-byte base64url, not returned in API responses
- **Audit Trail**: All create/accept/revoke actions logged for compliance
- **Expiry Handling**: Expired invitations rejected; proactive cleanup can be added via cron

## GDPR & Privacy Operations

### Data Subject Requests (DSR)
- **Export**: `POST /api/dsr/export` → User data export as JSON
- **Delete**: `POST /api/dsr/delete` → Anonymize user (soft delete)
- **Processing**: Manual review in follow-up (auto-processing later)

### Retention Management
- **Manual sweep**: `POST /api/admin/retention/run` (admin only)
- **Assets**: Configurable via `RETENTION_EXPORT_ASSETS_DAYS` (default: 365 days)
- **Jobs**: Configurable via `RETENTION_JOBS_DAYS` (default: 90 days)
- **Automation**: Platform scheduler (Vercel Cron) in future PR

### Audit Logging
- **Coverage**: DSR operations, retention sweeps, export downloads
- **Correlation**: Request ID tracking for distributed tracing
- **Storage**: Database audit_logs table with JSON metadata

### Cookie Consent
- **Endpoint**: `POST /api/consent` → Set consent=accepted|rejected cookie
- **Duration**: 1 year cookie expiry
- **Integration**: Gates analytics/tracking (placeholder for future)

## Observability & Analytics

### Error Monitoring (Sentry)
- **Setup**: Configured via `SENTRY_DSN` environment variable
- **Privacy**: PII automatically stripped from error events
- **Tagging**: Request IDs, locale, pseudo user IDs (hashed)
- **Sampling**: Configurable via `SENTRY_TRACES_SAMPLE_RATE` and `SENTRY_PROFILES_SAMPLE_RATE`
- **CSP**: Egress allowed to `https://o*.ingest.sentry.io`

### Product Analytics (PostHog Server SDK)
- **Consent Gating**: Only tracks when `consent=accepted` cookie is set
- **Privacy**: User IDs are hashed, no PII (emails/names) sent
- **Events Tracked**:
  - `report_computed` - After successful emission calculations
  - `report_exported` - After PDF export generation
- **Metadata**: Request ID, locale, report/asset IDs, performance metrics
- **Configuration**: 
  - `POSTHOG_KEY` - Project API key
  - `POSTHOG_HOST` - Instance URL (default: app.posthog.com)
  - `POSTHOG_ENABLED` - Global enable/disable (default: true)
- **CSP**: Egress allowed to `https://app.posthog.com`

### Request Correlation
- **x-request-id** header propagated to Sentry and PostHog
- **Distributed tracing** across API calls, error events, and analytics
- **Audit trail** linkage for GDPR compliance

### Factor Administration Operations

#### Factor Import Procedure
1. **Data Preparation**: Prepare factor JSON with source metadata, version, and factors array
2. **Authentication**: Ensure admin/owner authentication for `POST /api/admin/factors/import`
3. **Import Execution**: 
   - Factors are imported as new version under specified source
   - Existing factors remain untouched (additive only)
   - Import job tracks success/failure counts
4. **Verification**: Check audit logs for `FACTORS_IMPORT` entries
5. **Rollback Strategy**: Version-based rollback by updating factor queries to use previous version

#### Organization Override Management
1. **Creation**: Use `POST /api/orgs/{orgId}/factors/override` with proper authentication
2. **Precedence**: Org overrides take priority over country/EU factors within date ranges
3. **Immutability**: Overrides are immutable; supersede by creating new overrides with later `validFrom`
4. **Audit Trail**: All override operations logged with `FACTOR_OVERRIDE_CREATE` action

#### Factor Resolution Priority
1. **Org Override** (exact kind, date within valid range)
2. **Country Factor** (org's countryCode)
3. **EU Fallback** (default for all missing country factors)

#### Feature Flags
- `FEATURE_FACTORS_WRITE=true` - Enables factor import endpoints (can be disabled in production)
- Set to `false` to make factor database read-only

### PDF Engine Flag
- `PDF_ENGINE=mock` (default) uses a lightweight placeholder generator.
- `PDF_ENGINE=react-pdf` enables React-PDF. If rendering fails at runtime, the server logs a warning and falls back to mock.
- Troubleshooting:
  - Keep React-PDF import **dynamic** to avoid build-time bundling issues.
  - Ensure the route runs on Node.js runtime (not Edge).

### Export Caching & Retention
- PDFs are cached based on content hash (report data + framework versions + factors).
- Local development: Files stored in `.data/exports` (gitignored).
- Production: Configurable storage driver (local/S3).
- Signed URLs: 24h expiration (configurable via `SIGNED_URL_TTL_SECONDS`).
- Retention policy: Assets are kept indefinitely; add cleanup jobs for production.
